### NEW CLASS (priority): UNNUMBERED_SORT_KEY (Infinity) leaks into numeric comparison/aggregation in consumers

The sortKey-0 fix made unnumbered chapters `sortKey = Infinity` (correct for sorting), but consumers that do `Math.max`/`>`/`<` on sortKey without a `Number.isFinite` guard now misbehave. Convention already exists (sibling sites guard with `Number.isFinite`); these 4 sites missed it:

1. **SEVERE - `apps/extension/src/background/chapter-cache.ts` ~L238** (listChaptersWithTabFallback standard SW-fetch path): `Math.max(...chapters.map(c=>c.sortKey))` with no finite guard -> writes `latestChapterNumber: Infinity` -> JSON-serializes to null on backup export -> import schema `z.number().finite()` REJECTS it -> whole manga + progress + bookmarks dropped as PARENT_SKIPPED on restore = **silent title loss**. Also breaks completed-status. FIX: mirror `applyUpdateCheckResult` (database.ts ~905-912) - latest from max FINITE sortKey, guard the persisted write with Number.isFinite.
2. `apps/extension/src/handlers/updates-sources.ts` ~L342 (newChaptersFor): `c.sortKey > sinceKey` where sinceKey finite -> Infinity always > sinceKey -> oneshot shows as "new/unread" forever. FIX: unnumbered chapters should be "new" by id-not-seen, not by sortKey>.
3. `apps/extension/src/handlers/updates-sources.ts` ~L115 (checkUpdates latest reduce): Infinity wins max -> latestChapterId pinned to a oneshot, can mask later real new chapters. FIX: max over FINITE sortKeys.
4. `apps/extension/src/database.ts` ~L601 (remapExternalChapterProgress): guard is `extChapter.sortKey > 0` (meant to exclude old 0 sentinel); `Infinity > 0` passes -> `find(c=>c.sortKey===Infinity)` matches wrong chapter, skips pathname fallback. FIX: change guard to `Number.isFinite(extChapter.sortKey)`.
   CLOSE THE CLASS: add a shared `latestNumberedChapter(chapters)` helper (max finite sortKey) used at sites 1+3, and a meta-test tripwire scanning `apps/extension/src` for `Math.max(...sortKey...)` / unguarded `.sortKey >`/`latestChapterNumber:` from a raw sortKey.
